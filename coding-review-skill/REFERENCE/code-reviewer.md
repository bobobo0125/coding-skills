## 审查清单

**代码质量：**
- 关注点分离清晰？
- 正确的错误处理？
- 类型安全（如果适用）？
- 遵循DRY原则？
- 边界情况处理？

**架构：**
- 设计决策合理？
- 可扩展性考虑？
- 性能影响？
- 安全问题？

**测试：**
- 测试实际测试逻辑（不是mock）？
- 边界情况覆盖？
- 需要时是否有集成测试？
- 所有测试通过？

**需求：**
- 满足所有计划需求？
- 实现符合规格？
- 没有范围蔓延？
- 破坏性变更已记录？

**生产就绪：**
- 迁移策略（如果 schema 变更）？
- 向后兼容性考虑？
- 文档完整？
- 没有明显的 bug？

**对于每个问题：**
- 文件:行号引用

- 为什么重要
- 如何修复（如果不是很明显）

### 建议
[代码质量、架构或流程的改进建议]

### 评估

**可以合并？** [是/否/修复后]

**理由：** [1-2句技术评估]

## 关键规则

**应该：**
- 按实际严重程度分类（不是所有问题都是 Critical）
- 具体说明（文件:行号，不要模糊）
- 解释为什么问题重要
- 认可优点
- 给出明确的判断

**不应该：**
- 不检查就说"看起来不错"
- 把小问题标记为 Critical
- 对未审查的代码给出反馈
- 模糊（如"改进错误处理"）
- 避免给出明确的判断
