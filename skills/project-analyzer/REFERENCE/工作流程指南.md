# 项目分析器 - 渐进式分析工作流程

本指南扩展了项目分析器技能，提供基于Kubernetes源代码分析方法的系统化深入分析 methodology。

## 分析模式

### 🚀 快速评估模式（6个主题）
**使用场景**：快速评估、基本了解
**时间**：10-15分钟
**主题**：1, 3, 4, 9, 10, 12

### 📊 标准分析模式（12个主题）
**使用场景**：一般项目分析、架构审查
**时间**：30-45分钟
**主题**：所有标准主题

### 🔬 深度分析模式（20个主题）
**使用场景**：技术项目、源代码分析、内部机制理解
**时间**：1-2小时
**主题**：所有标准主题 + 高级主题

## 渐进式文档创建流程

### 每个主题的分析周期

对每个主题执行以下步骤：

1. **📖 分析阶段**
   - 收集信息和数据
   - 创建必要的图表
   - 分析核心内容

2. **💾 保存阶段**
   - 创建独立主题文档：`topics/XX-[主题名].md`
   - 保存到项目目录
   - 记录时间戳和元数据

3. **📊 更新阶段**
   - 更新主综合报告中的相应部分
   - 更新进度跟踪文档
   - 添加文档间的链接

4. **📢 报告阶段**
   - 向用户报告完成状态
   - 显示关键发现摘要
   - 提供文档位置链接

5. **➡️ 继续阶段**
   - 自动移动到下一个主题
   - 无需用户确认

### 文档创建时机

```
主题开始 → 分析完成 → 立即创建文档 → 更新进度 → 报告 → 下一个主题
     ↓              ↓                    ↓              ↓            ↓
   开始       内容准备             文件保存          状态更新       用户通知
```

### 文档保存策略

**立即保存原则**：
- 每个主题分析完成后立即创建独立文档
- 不要等待整个分析完成
- 防止数据丢失
- 提供实时进度可见性

**保存内容**：
- 详细的主题分析内容
- 相关的Mermaid图表
- 关键发现和洞察
- 参考资源链接
- 分析元数据（时间、时长、置信度）

## 渐进式分析策略

### 第一阶段：基础（主题1-3）
**目标**：了解项目基础和结构
**方法**：
- 从GitHub API数据收集开始
- 建立项目组织的心理模型
- 识别主要编程语言和框架

### 第二阶段：架构（主题4-6）
**目标**：掌握设计和实现质量
**方法**：
- 映射核心功能和用户流程
- 理解架构模式
- 评估代码质量和测试方法

### 第三阶段：生态（主题7-12）
**目标**：评估项目健康状况和使用情况
**方法**：
- 审查文档质量
- 分析社区活跃度
- 形成实用建议

### 第四阶段：技术深度分析（主题13-20）
**目标**：理解内部机制（可选）
**方法**：
- 追踪源代码执行路径
- 分析关键算法和数据结构
- 评估性能和安全性特征
- 记录实现细节

## 代码分析方法

### 入口点分析
1. **识别主要入口点**：
   - 应用的主 `main()` 函数
   - Web服务的API端点
   - SDK的库导出
   - CLI命令处理器

2. **追踪初始化序列**：
   - 配置加载
   - 依赖注入
   - 服务启动
   - 资源分配

### 数据结构映射
1. **核心实体**：识别主要数据结构
2. **关系**：映射依赖和关联
3. **状态管理**：理解状态转换
4. **数据流**：跟踪数据在系统中的流动

### 控制流分析
1. **请求处理**：跟踪典型请求生命周期
2. **错误处理**：理解错误传播
3. **资源管理**：跟踪分配/清理
4. **并发**：分析并行处理模式

### 接口分析
1. **API契约**：记录接口规范
2. **通信模式**：理解组件交互
3. **协议分析**：检查通信协议
4. **数据序列化**：分析数据格式处理

## 渐进式深入步骤

### 级别1：表面理解
- 阅读README和文档
- 检查目录结构
- 运行基本功能
- 审查示例

### 级别2：结构分析
- 映射模块依赖
- 识别关键组件
- 理解配置系统
- 审查测试方法

### 级别3：实现细节
- 追踪核心算法
- 分析数据结构
- 审查错误处理
- 学习优化技术

### 级别4：高级主题
- 性能特性
- 安全机制
- 扩展点
- 集成模式

## Mermaid图表策略

### 何时创建图表
1. **早期分析**：简单的结构图
2. **中期分析**：添加序列图和流程图
3. **深度分析**：全面的技术图表

### 图表选择指南

| 分析阶段 | 图表类型 | 目的 |
|---------------|--------------|---------|
| 基础 | 模块图 | 展示结构 |
| 架构 | 架构图 | 展示层次 |
| 实现 | 序列图 | 展示流程 |
| 深度分析 | 数据流图、状态图 | 展示机制 |

## 代码引用格式

### 标准格式
```
位置: pkg/module/file.go:123
函数: ProcessRequest()
上下文: API服务器中的HTTP请求处理
```

### 渐进式详细级别
1. **基础**：模块和文件名
2. **详细**：文件名和行号
3. **全面**：函数签名和上下文

## 分析延续框架

### 交接策略
当需要继续分析时：
1. **当前状态**：记录已完成的主题
2. **下一个优先级**：识别剩余领域
3. **入口点**：为继续分析提供具体位置
4. **上下文**：分享关键发现和洞察

### 重新分析触发条件
- 主要版本更新
- 架构变更
- 性能问题
- 安全漏洞

## 质量指标

### 分析完整性检查清单
- [ ] 覆盖所有必需主题
- [ ] 用证据记录关键发现
- [ ] 在适当处包含图表
- [ ] 为技术声明提供代码引用
- [ ] 包含实用建议
- [ ] 注明来源

### 技术深度指标
- **表面**：基本功能描述
- **中等**：架构理解
- **深度**：带代码引用的实现细节
- **专家**：性能、安全和优化分析

## 渐进式分析流程示例

### 初始分析（标准模式）
```
1. 快速概览（15分钟）
2. 功能映射（10分钟）

```

```

4. 未来发展轨迹
5. 研究论文准备
```

- 代码搜索：查找具体实现

### 分析工具

- 依赖分析器：模块关系
- 性能分析工具：性能数据（如有）

## 最佳实践

### 时间管理
- 为每个主题设置明确的时间限制
- 根据用户需求优先排序
- 使用渐进式深入方法
- 记录假设和限制

### 准确性和验证
- 从多个来源交叉验证信息
- 尽可能根据实际代码测试声明
- 记录不确定领域
- 为评估提供置信度

### 沟通
- 尽可能使用清晰、非技术语言
- 在单独章节提供技术细节
- 为复杂概念包含可视化图表
- 提供实用、可操作的建议
