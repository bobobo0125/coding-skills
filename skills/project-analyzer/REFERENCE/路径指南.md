# 项目分析器 - 路径指南

## 🎯 核心原则

**分析文档始终保存在被分析项目的 `ai-analysis-docs/` 目录中**

## 📁 目录结构规则

### 正确的路径结构

```
[被分析的项目]/
├── [项目的原始文件和目录]
├── [其他项目文件]
└── ai-analysis-docs/              ← 在此处创建分析文档
    ├── changelog.md
    ├── [项目名]-analysis.md
    ├── [项目名]-progress-tracking.md
    ├── analysis-todo.md
    ├── topics/
    │   ├── 01-project-basic-info.md
    │   ├── 02-project-structure.md
    │   └── ...
    └── assets/
```

## 📍 具体示例

### 示例1：分析Kubernetes项目

**用户命令**：`分析 /Users/ccc/work/todo/kubernetes`

**文档保存位置**：
```
/Users/ccc/work/todo/kubernetes/ai-analysis-docs/
├── changelog.md
├── kubernetes-analysis.md
├── kubernetes-progress-tracking.md
├── analysis-todo.md
├── topics/
│   ├── 01-project-basic-info.md
│   ├── 02-project-structure.md
│   └── ...
└── assets/
```

### 示例2：分析当前目录

**用户命令**：`分析 .` 或 `分析当前目录`

**文档保存位置**：
```
./ai-analysis-docs/
├── changelog.md
├── [当前目录名]-analysis.md
├── [当前目录名]-progress-tracking.md
├── analysis-todo.md
├── topics/
└── assets/
```

### 示例3：分析相对路径项目

**用户命令**：`分析 ../my-project`

**文档保存位置**：
```
../my-project/ai-analysis-docs/
├── changelog.md
├── my-project-analysis.md
├── my-project-progress-tracking.md
├── analysis-todo.md
├── topics/
└── assets/
```

## ❌ 错误示例

### 不要使用这样的路径：

```
❌ /Users/ccc/.openclaw/workspace/[项目名]/ai-analysis-docs/
❌ ~/analysis-results/[项目名]/
❌ /tmp/analysis/[项目名]/
❌ [任何项目外的其他位置]
```

## ✅ 正确示例

### 应该使用这样的路径：

```
✅ /Users/ccc/work/todo/kubernetes/ai-analysis-docs/
✅ ./ai-analysis-docs/ (当前目录)
✅ ../my-project/ai-analysis-docs/ (相对路径)
✅ [项目路径]/ai-analysis-docs/ (通用形式)
```

## 🔍 路径解析规则

### 绝对路径
- 输入：`/Users/ccc/work/todo/kubernetes`
- 输出：`/Users/ccc/work/todo/kubernetes/ai-analysis-docs/`

### 相对路径
- 输入：`./my-project`
- 输出：`./my-project/ai-analysis-docs/`

### 当前目录
- 输入：`.` 或 `当前目录`
- 输出：`./ai-analysis-docs/`

## 🎯 为什么要这样设计？

### 1. **强项目关联**
分析文档与项目紧密关联，放在项目内便于：
- 项目版本控制
- 团队共享分析结果
- 分析与代码同步更新

### 2. **易于管理**
- 一个项目，一个分析目录
- 分析结果随项目移动
- 项目删除时分析文档自动删除

### 3. **版本控制友好**
- 可以将分析文档加入Git
- 分析历史与项目历史一起管理
- 团队协作可以共享分析洞察

### 4. **路径简洁**
- 不需要额外的工作区目录
- 分析文档始终相对于项目位置
- 减少路径混淆的可能性

## 📝 实现注意事项

### 创建目录时
```bash
# 确保在被分析的项目内创建
mkdir -p [项目路径]/ai-analysis-docs/topics
mkdir -p [项目路径]/ai-analysis-docs/assets
```

### 保存文件时
```bash
# 所有文件都保存在项目内
changelog_path = "[项目路径]/ai-analysis-docs/changelog.md"
main_report_path = "[项目路径]/ai-analysis-docs/[项目名]-analysis.md"
topics_dir = "[项目路径]/ai-analysis-docs/topics/"
```

### 报告路径时
```bash
# 向用户报告时，使用完整路径
echo "分析文件已保存：[项目路径]/ai-analysis-docs/[项目名]-analysis.md"
```

## 🔄 迁移指南

如果以前使用其他路径结构，可以按以下方式迁移：

### 旧结构 → 新结构
```
旧：/Users/ccc/.openclaw/workspace/kubernetes/ai-analysis-docs/
新：/Users/ccc/work/todo/kubernetes/ai-analysis-docs/

迁移命令：
mv /Users/ccc/.openclaw/workspace/kubernetes/ai-analysis-docs/ \
   /Users/ccc/work/todo/kubernetes/ai-analysis-docs/
```

## 💡 最佳实践

1. **一致性**：始终将分析文档保存在项目的 `ai-analysis-docs/` 目录中
2. **忽略规则**：考虑是否在项目的 `.gitignore` 中忽略分析文档
3. **团队共享**：如果团队共享分析结果，将分析文档加入版本控制
4. **定期清理**：项目删除时分析文档随之删除
