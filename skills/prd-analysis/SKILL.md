# 需求分析技能

## 目的

诊断需求层面的问题，将模糊意图转化为经过验证、可测试的需求。引导开发者从"我想构建X"到清晰的问题陈述、优先排序的需求、明确的约束和有限的作用域——在任何设计或实施之前完成。

---

## 核心目标

**主要目标**：生成一份经过验证的需求文档，其中每个需求都是可测试的、有作用域的、并基于真实问题。

**成功标准**（必须全部满足）：

1. ✅ **问题已阐述**：存在不引用任何解决方案或技术的清晰问题陈述
2. ✅ **需求可测试并有ID**：每个需求都有验收标准和唯一的 `R-NN` ID（如R-01、R-02）；ID按顺序，无间隙或重复
3. ✅ **约束已盘点**：真实约束（预算、时间、技能、依赖）与假设分开列出
4. ✅ **作用域已界定**：V1边界明确，列出待延期项目并记录重新考虑的触发条件
5. ✅ **用户已确认**：用户明确批准了经过验证的需求（说"批准"、"看起来不错"、"继续"或等效表达）
6. ✅ **文档已持久化**：需求文档按规范写入 `docs/requirements-planning/<主题>.md` 并提交

**验收测试**：不熟悉项目的人能否阅读需求文档并理解问题、谁有这个问题、"完成"是什么样子、以及什么是超出范围的——无需提出澄清问题？

---

## 范围边界

**本技能处理**：

- 模糊意图 → 经过验证的需求文档
- 问题发现和阐述
- 需求澄清和验收标准编写
- 约束盘点和假设映射
- 作用域定义和优先级排序（V1 vs 后续）
- 需求质量评估（清晰度、具体性、完整性）

**本技能不处理**：

- 设计和架构决策（使用 `design-solution`）
- 实现规划或任务列表（使用实现规划技能）
- 编写代码（使用开发技能）
- 代码审查（使用 `review-code`）
- 技术选型（提及约束，但技术选择属于设计阶段）

**交接点**：当需求经过验证（满足所有成功标准）时，移交给 `design-solution` 进行设计探索，如果设计很简单则移交给实现规划。

---

## 使用场景

- **新项目启动**：开发者说"我想构建X"——在任何设计之前提取真实问题并验证需求
- **功能请求分类**：利益相关者提交模糊的功能请求——澄清问题、作用域和验收标准
- **防止范围蔓延**：项目不断扩大——应用优先级排序和明确的V1边界
- **需求验证**：现有需求感觉不完整——诊断它们处于什么状态并推进它们
- **约束发现**：项目中后期出现意外的阻碍——在影响工作之前揭示隐藏的约束和假设

## 行为

### 交互策略

- **默认值**：从输入推断当前状态；从最早未解决的状态开始
- **选择选项**：一次一个问题；尽可能提供 `[A][B][C]` 选项
- **确认**：在退出到设计交接之前；在编写需求文档之前

### HARD-GATE：验证前不设计

```
不要提出架构、选择技术、创建设计，
或在需求验证之前编写代码。

需求描述的是问题和需求，而不是解决方案。
如果需求提及技术，那就是伪装成需求的解决方案。
```

### 阶段0：分类

**开始时宣布：** "我正在使用需求分析技能来验证需求，然后再进行任何设计或实现。"

对输入进行快速质量评估：

| 维度 | 分数范围 | 检查内容 |
| :--- | :--- | :--- |
| **清晰度** | 0–100% | 是否有唯一无歧义的解释？术语是否定义？ |
| **具体性** | 0–100% | 是否提供上下文？是否提及成功标准？作用域是否有界？ |
| **完整性** | 0–100% | 是否存在所有必要信息？是否提及约束？ |

**决策矩阵**：

| 总分 | 行动 |
| :--- | :--- |
| < 40% | 从状态RA0（问题发现）开始 |
| 40–70% | 识别最早的问题状态并从那里开始 |
| > 70% | 快速验证并专注于差距 |

### 阶段1：诊断 — 识别当前状态

按顺序推进状态。不要跳过状态——如果问题不清晰，不要讨论需求。

#### 状态RA0：没有问题陈述

**症状**：从解决方案开始（"我想构建X"）；功能列表没有问题基础；"每个人都需要这个"推理；无法阐明谁有什么问题。

**关键问题**（一次问一个）：

- "什么具体的挫折或痛点导致了这个想法？"
- "人们（或你）现在用什么来代替？"
- "如果这不存在会发生什么？谁受影响，怎么受影响？"
- "如果你是用户，是什么触发你现在思考这个？"

**干预措施**：

- 问题陈述简报：记录谁有什么问题以及为什么重要
- "五个用户"测试：说出5个会受益的具体人名（即使其中一个是你自己）
- 待完成工作："当我在[情况]时，我想要[动机]，这样我就可以[结果]"
- 问题考古：追溯回特定的挫折事件

**退出标准**：存在不提及任何解决方案或技术的问题陈述。

#### 状态RA1：解决方案优先思维

**症状**：在问题清晰之前选择技术（"需要数据库"、"应该使用React"）；需求描述实现而不是需求；无法在不引用技术的情况下解释需求。

**关键问题**：

- "这个功能背后的需求是什么？"
- "如果没有那种技术，你仍然需要什么？"
- "你是在解决你自己的问题还是在复制别人的解决方案？"
- "这个技术是必需的（约束）还是只是熟悉的（偏好）？"

**干预措施**：

- "移除解决方案"练习：用任何实现词汇描述需求
- 功能提取：将每个需求重写为"系统必须[动词]..."，不含技术术语
- 约束vs偏好区分：将硬约束与默认值分开

**退出标准**：每个需求都描述需求，而不是实现。技术引用被移除或明确记录为真实约束。

#### 状态RA2：模糊需求

**症状**：无法描述"完成"是什么样子；基于形容词的需求（"快速"、"简单"、"直观"）；无法测试的需求；"用户应该能够..."但没有具体说明。

**关键问题**：

- "你能给出一个具体的场景示例吗？"
- "令人失望的实现和很棒的实现分别是什么样子？"
- "什么是最小限度满足这个需求的东西？"
- "你怎么知道这个需求是否被满足？"

**干预措施**：

- 验收场景编写："给定[上下文]，当[行动]，那么[结果]"
- 具体性阶梯：具体是谁？具体做什么？具体什么时候？
- "完成的样子像..."练习：描述满足需求的最小东西
- 可测试性检查：如果你不能为它写测试，说明你还不理解它

**退出标准**：每个需求都有验收标准。不再存在仅用形容词的需求。

#### 状态RA3：隐藏约束

**症状**：后期出现意外的依赖；没有明确的约束清单；假设被当作事实；讨论中发现阻碍。

**关键问题**：

- "存在什么外部依赖？"
- "你实际拥有什么资源、技能和时间？"
- "什么会杀死这个项目，如果它被证明是真的？"
- "你假设什么是真的但还没有验证？"

**干预措施**：

- 约束清单：将预算、时间、技能、依赖、集成列为**真实约束**
- 假设映射：将已验证的事实与未验证的假设分开
- 风险预演："6个月后它失败了。为什么？"
- 依赖发现：什么必须先存在才能使这个工作？

**退出标准**：存在带有真实约束与假设明确分离的约束清单。

#### 状态RA4：范围蔓延

**症状**：每个功能感觉同等重要；没有V1边界；"既然我们在做..."的添加；需求扩展快于满足。

**关键问题**：

- "如果你只能交付3样东西，它们是什么？"
- "什么可以削减但仍能解决核心问题？"
- "最小的有用版本是什么？"
- "什么会触发重新考虑延期的项目？"

**干预措施**：

- 削减优先方法：从全部移除开始，只添加必要的东西
- 强制排序：严格排序，无并列
- 行走骨架：确定最薄的有用版本
- MoSCoW：必须有 / 应该有 / 可以有 / 不会有
- 延期功能清单，包含重新考虑的明确触发条件

**退出标准**：V1边界明确。列出延期项目及触发条件。作用域内有什么没有歧义。

#### 状态RA5：需求已验证

**指标**：

- 问题陈述存在，无解决方案引用
- 每个需求都有可测试的验收标准
- 约束已盘点（真实vs假设）
- 作用域有明确的V1定义
- 可以向不熟悉的人解释并让他们理解需求

**下一步**：将经过验证的需求文档移交给 `design-solution`。

### 阶段2：验证 — 确认用户

1. 以结构化格式呈现需求摘要（见输出部分）
2. **请求明确批准**："这些需求准确捕捉了你的需要吗？"
3. **如有需要则迭代**：如果发现差距，返回相关状态

### 阶段3：持久化 — 编写文档

1. **确定输出位置**：
   - 默认：`docs/requirements-planning/<主题>.md`
   - 兼容性：使用 `docs/requirements/<主题>.md` 的现有项目可暂时保留该路径
   - 如果用户偏好不同位置，询问用户
2. 按照输出模板编写文档
3. 提交到版本控制
4. 宣布完成和交接

## 输入与输出

### 输入

- **用户请求**：模糊的想法、功能请求、问题陈述或要验证的现有需求。
- **项目上下文**：现有代码库、文档、约束（在探索中发现）。
- **用户响应**：诊断问题的答案、对评估的反馈、对经过验证需求的批准。

### 输出

#### 经过验证的需求文档

```markdown
# [主题] 需求

**日期：** YYYY-MM-DD
**状态：** 已验证
**批准者：** [用户名或"用户"]

## 问题陈述

[谁有什么问题以及为什么重要。不包含解决方案或技术引用。]

## 需求层级

### 必须有 (V1)

| ID | 需求 | 验收标准 |
| :--- | :--- | :--- |
| R-01 | [可测试的需求] | 给定[X]，当[Y]，那么[Z] |
| R-02 | [可测试的需求] | [可衡量标准] |

### 应该有 (V1如果时间允许)

| ID | 需求 | 验收标准 |
| :--- | :--- | :--- |
| R-03 | [可测试的需求] | [标准] |

### 可以有 (V1后)

| ID | 需求 | 重新考虑的触发条件 |
| :--- | :--- | :--- |
| R-04 | [延期需求] | [何时重新审视] |

## 约束清单

### 真实约束（已验证）

- [预算、时间、技能、依赖、集成]

### 假设（未验证）

- [需要验证的假设，以及验证计划]

## 作用域定义

- **在范围内 (V1)：** [明确列表]
- **不在范围内：** [明确列表]
- **行走骨架：** [最薄的有用版本]

## 开放问题

- [任何剩余未知数及解决计划]
```

## 限制

### 硬边界

- **验证前不设计**：在需求验证之前不要提出架构、选择技术或创建设计。
- **不跳过状态**：如果问题不清晰（RA0），不要讨论需求（RA2）或范围（RA4）。按顺序推进。
- **一次一个问题**：不要在一条消息中用多个诊断问题淹没用户。
- **开发者决定**：诊断、提问和引导——开发者对优先级和范围做出最终决定。
- **不接受模糊验收**：不接受基于形容词的需求（"快速"、"直观"）作为完整的。推动可测试的标准。
- **不使用解决方案语言**：需求必须描述问题和需求，而不是实现。标记并重写伪装成需求的解决方案。

### 技能边界（避免重叠）

**不要做这些（其他技能处理）**：

- **设计和架构**：提出解决方案、选择技术、创建设计文档 → 使用 `design-solution`
- **实现规划**：创建任务列表、文件路径、代码结构 → 使用实现规划技能
- **编写代码**：编写任何实现代码 → 使用开发技能
- **代码审查**：审查现有代码 → 使用 `review-code`
- **技能改进**：改进技能文档 → 使用 `refine-skill-design`

**何时停止和交接**：

- 用户说"批准"、"看起来不错"、"继续" → 需求已验证，移交给 `design-solution`
- 用户问"我们应该如何设计这个？" → 移交给 `design-solution`
- 用户问"你能写代码吗？" → 需求完成，移交给开发工作流（如果跳过设计则建议）
- 用户说需求"足够好" → 确认满足所有成功标准，然后交接

## 自检

### 核心成功标准（必须全部满足）

- [ ] **问题已阐述**：存在不含解决方案或技术引用的清晰问题陈述
- [ ] **需求可测试并有ID**：每个需求都有验收标准和唯一的 `R-NN` ID；ID按顺序，无间隙或重复
- [ ] **约束已盘点**：真实约束与假设分开
- [ ] **作用域已界定**：V1边界明确，延期项目和触发条件已记录
- [ ] **用户已确认**：用户明确批准了经过验证的需求
- [ ] **文档已持久化**：需求文档写入 `docs/requirements-planning/<主题>.md` 并提交

### 过程质量检查

- [ ] **状态未跳过**：按诊断状态顺序推进（没有从RA0跳到RA4）
- [ ] **一次一个问题**：没有用每条消息的多个问题淹没用户
- [ ] **无解决方案语言**：需求描述需求，而不是实现
- [ ] **开发者决定**：引导和提问，但开发者做出最终优先级和范围决定
- [ ] **形容词已解决**：没有剩余模糊的仅用形容词的需求（"快速"、"直观"、"简单"）
- [ ] **假设已分开**：未验证的假设与真实约束分开标记
- [ ] **ID已检查**：所有需求ID遵循 `R-NN` 格式，唯一，并在文档中的交叉引用中使用

### 健康检查问题

在宣布需求已验证之前，问自己：

1. 如果这个项目失败，最可能与需求相关的原因是什么？
2. 开发者是在构建他们需要的还是他们知道如何构建的？
3. 什么明确不在范围内？
4. 范围在实际约束下可实现吗？
5. 什么假设还没有验证？
6. 不熟悉的人能否正确解释需求？

### 验收测试

**不熟悉项目的人能否阅读需求文档并理解问题、谁有这个问题、"完成"是什么样子、以及什么是超出范围的——无需提出澄清问题？**

如果否：需求不完整。返回最早未解决的状态。

如果是：需求已验证。持久化并移交给 `design-solution`。
