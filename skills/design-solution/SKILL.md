# 技术方案设计技能

## 目的

将经过验证的需求文档转化为可执行的技术方案。引导开发者从清晰的"做什么"到明确的"怎么做"——包括架构设计、技术选型、接口定义、数据模型、业务流程等。

---

## 核心目标

**主要目标**：生成一份经过评审、可直接指导开发的技术方案文档。

**成功标准**（必须全部满足）：

1. ✅ **需求可追溯**：每个技术决策都能追溯到对应的需求 ID
2. ✅ **架构已定义**：整体架构、新增模块、模块职责清晰
3. ✅ **技术已选型**：关键技术栈有明确的选型理由
4. ✅ **接口已设计**：核心接口定义完整（路径、请求、响应）
5. ✅ **数据已建模**：数据库表结构、ER 图完整
6. ✅ **流程已绘制**：核心业务流程有时序图说明
7. ✅ **方案已评审**：技术方案经过团队评审或用户确认

**验收测试**：开发人员能否直接根据技术方案编写代码，无需额外技术决策？

---

## 范围边界

**本技能处理**：

- 架构设计（整体架构、分层结构）
- 技术选型（语言、框架、中间件）
- 模块设计（模块划分、类结构）
- 接口设计（REST API、协议）
- 数据建模（数据库表、ER 图）
- 业务流程（时序图、活动图）
- 安全设计（认证、授权、数据安全）
- 性能设计（指标、优化策略）
- 兼容性设计（版本策略、灰度方案）
- 部署设计（部署架构、配置）
- 测试策略（单元测试、集成测试）
- 风险评估（风险识别、应对措施）

**本技能不处理**：

- 需求分析（使用 `analyze-requirements`）
- 编写代码（使用 `coding-skill`）
- 代码审查（使用 `coding-review-skill`）

**交接点**：当技术方案评审通过后，移交给 `coding-skill` 进行开发。

---

## 使用场景

- **新功能开发**：需求已验证，需要设计技术实现方案
- **功能修改**：现有功能需要重构或扩展
- **架构调整**：系统架构需要优化或调整
- **技术调研**：需要评估新技术方案的可行性

## 行为

### 交互策略

- **默认值**：从需求文档提取设计要点
- **确认关键决策**：重大技术选型需与用户确认
- **迭代优化**：根据反馈调整方案

**开始时宣布：** "我正在使用技术方案设计技能，将需求转化为可执行的技术方案。"

2. 分析需求中的关键点（功能点、性能要求、安全要求等）

## 限制

### 硬边界

- **需求驱动设计**：每个技术决策必须有对应的需求支撑
- **不跳过评审**：技术方案必须经过评审才能进入开发
- **可追溯**：设计变更必须记录变更原因

### 技能边界（避免重叠）

**不要做这些（其他技能处理）**：

- **需求分析**：需求分析、需求验证 → 使用 `analyze-requirements`
- **编写代码**：编写任何实现代码 → 使用 `coding-skill`
- **代码审查**：审查现有代码 → 使用 `coding-review-skill`

**何时停止和交接**：

- 技术方案评审通过 → 移交给 `coding-skill`
- 用户要求修改需求 → 移交给 `analyze-requirements`
- 需要技术调研 → 在方案中说明调研计划

## 自检

### 核心成功标准（必须全部满足）

- [ ] **需求可追溯**：每个设计要点都有对应的需求 ID
- [ ] **架构已定义**：整体架构、新增模块清晰
- [ ] **技术已选型**：关键技术栈有选型理由
- [ ] **接口已设计**：核心接口定义完整
- [ ] **数据已建模**：数据库表结构完整
- [ ] **流程已绘制**：核心业务流程有时序图
- [ ] **方案已评审**：经过评审或用户确认

### 过程质量检查

- [ ] **需求完整性**：是否覆盖所有需求
- [ ] **技术合理性**：技术选型是否合理
- [ ] **可实施性**：方案是否可以直接指导开发
- [ ] **风险识别**：是否识别主要风险
- [ ] **向后兼容**：是否考虑兼容性

### 健康检查问题

在宣布技术方案完成之前，问自己：

1. 开发人员能否直接根据这个方案编写代码？
2. 是否有任何技术决策没有需求支撑？
3. 是否遗漏了重要的异常场景？
4. 性能指标是否在合理范围内？
5. 是否有无法应对的风险？

### 验收测试

**开发人员能否直接根据技术方案编写代码，无需额外技术决策？**

如果是：技术方案完成。提交评审并移交给 `coding-skill`。

如果否：补充缺失的设计内容。
